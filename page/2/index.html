那场上海Android交流会

这场盛会的相关信息

  • 地点：上海
  • 发起人：stormzhang
  • 主题：Android技术交流会
  • 天气：暴雨(天气是暴雨哦)

  • 天朝的奇葩网络问题
  • Android组件化
  • BlockCanary
  • Android编译加速
  • Android内存优化浅谈

我的总结

这场交流会分享嘉宾都是来自开源社区的活跃者，例如：大头鬼、胡凯、Tomcat的猫、stormzhang、寒江不钓、别闹腾啊等；这次交流会总共有六个主题，如下：

天朝的奇葩网络问题

这是stormzhang分享的主题。这些问题主要是stormzhang在线上产品碰到的实战问题：

  • 购物车中的产品删除不了

    服务端API严格遵循RESTFul标准，手机端购物车删除产品时，使用的HTTP协议的DELETE方法；经过客户反馈和验证发现某些偏远的
    山区会出现这种情况。经过验证排查，将HTTP请求中的DELETE方法改为POST方法，可能是某些地方的运营商的服务器不支持HTTP的
    DELETE方法（可能是没有及时升级等）

    注意：这个具体原因，还得你细细推敲

  • 手机App加载网页时出现很多广告

    内容劫持

    解决方案：HTTP DNS

  • 用户连App也访问不了

    域名劫持或者DNS解析出错

    解决方案：HTTP DNS

Android组件化

随着业务的发展，项目会越来越大，项目中的代码耦合性也会越来越大，平时的开发中，改一行代码，都需要花很长时间去编译。 为此，寒江不钓所在的团队，考虑将整个项目拆分开来，拆成诸多的组件，以aar的方式依赖，这样降低了代码之间的偶尔性，也让团队分工更加明确。并在此基础上，编译时，加入增量检查步骤，没有改动的aar的模块不需要重新编译，这样加快了平时的调试速度，提升了团队的开发效率

BlockCanary

相信大家都知道或者用过Squre公司研发的久负盛名的LeakCanary. 而MarkZhai的这个BlockCanary借鉴于LeakCanary，是一个轻量的，非侵入式的性能监控组件，应用只需要简单地加几行，提供一些该组件需要的上下文环境就可以在使用应用的时候检测主线程上的各种卡顿问题，并通过组件提供的各种信息分析出原因并进行修复。具体的介绍可以参看：http://blog.zhaiyifan.cn/2016/01/16/BlockCanaryTransparentPerformanceMonitor/

Android编译加速

王光景（微博：别闹腾啊）针对现有Android编译慢的问题从原理了进行了描述，指出为什么慢，并提出可以从哪些方面提高速度，然后，在实战中，替换了一些编译工具（例如：Buck、LayoutCast、Instant Run等）来加快速度

Android内存优化浅谈

胡凯从五个方面对Android内存进行优化，并且他将此总结为：5R法则，分别是：

  • Reduce
    • Cache/Drawable
    • Load in demand
    • Bitmap-Scale/Format
  • Reuse
    • pools
    • inBitmaps
    • convertView
    • onDraw/for
  • Recyle
    • Inner class/container/static
    • Context
    • register/unregister
    • bitmap/cursor/webview
    • onTrimMemory
  • Refactor
    • arrayMaps/ no enum
    • avoid memory fragment
    • layout/overdraw
  • Revalue
    • largeHeap
    • multi-process
    • 3-party libs

大头鬼展现了他们团队在项目中如何运用RN，以实战经验分享了RN的优势以及不足

我竟然无意识地顿悟了

离开之前的创业团队已经有一段时间了。但此间，和朋友聊天，朋友总会关心地问一些相关的问题：

  • 创业的这段经历，你后悔过吗？
  • 创业付出的这段时间，你觉得值吗？
  • 之前创业的团队怎么样了？
  • 你为什么离开呢？
  • 以后你还会创业吗？
  • ……

而我每次都会坚定地回答到:

对于这次创业我从不后悔，可能不理智，可能没怎么想清楚，可能是错的，但是我觉得值。因为我学到了很多，就算我学到
的东西并没有多少，但最起码我尝试了。如果再来一次，让我去选择，我仍然会去尝试，后面如果时机成熟的话，我依然会去尝试。
我只是不想老来的时候，自己会惋惜和遗憾一些事情，怕的不是遇到困难，怕的是自己没有尝试一些想法和事情

不知道当你们听到这样的回答后会是什么反应，因为有时候，我也被我的这种回答懵到，我怕我说得太矫情，怕让别人感觉我是在装，总之，我感觉我的底气总不是那么太足。

但是最近在稻盛和夫的《活法》这本书里面看到这样一节：

我们人生的意义是什么？人生的目的在哪里？对于这个人生最基本的问题，我认为必须从正面回答。我的答案是：提升心性，磨练
灵魂。

人生在世，为欲所迷，为欲所困，可以说是我们人这种动物的本性。如果放任这种本性，我们就会无止境地追求财富、地位和名誉，
就会沉湎欲享乐。

当然，生活需要丰衣足食，自由活动需要相应的资金，希望立身出世也是进步的动力，这些都不应一概否定。

然而，上述的一切只能限于现世，再多也不能带往来世，今世的事情必须在今世清算完结。

如果人生有不灭之物，那就是“灵魂”。当死亡来临的时候，你在今世所创造的地位、名誉、财产就得统统放弃，只能带着
你的“灵魂”开始新的征程。

因此，如果有人问我：“你为何来到这世上？” 我会毫不含糊地回答：“是为了在死的时候，灵魂比生的时候更纯洁一点，或
者说带着更美好、更崇高的灵魂去迎接死亡。”

我们降临俗世，经受各种风浪的冲击，尝尽人间的苦乐，或幸福或悲伤，一直到呼吸停止之前，我们不懈地、顽强地努力奋
斗。这个人生的过程本身，就像磨练灵魂的砂纸，人们在磨练中提升心性，涵养精神，带着比降生时更高层次的灵魂离开人
世。我认为这是人生的目的，除此之外，人生再无别的目的。

今天比昨天做得更好，明天又比今天做得好，每一天都付出真挚的努力、不懈的工作、扎实的行动、诚恳的修道，在这样的
过程中就体现了我们人生的目的和价值。

人生不如意事十有八九。有时我们甚至怨恨神佛，为什么只让我经历那么多的苦难？但正是这些苦难才能磨练我们的灵魂，
提升我们的人格。把苦难看作考验、看作机会，我们需要这样来思考问题。

能够把考验看作是机会的人——只有这样的人，才能在有限的人生中让自己的生命绽放光彩。

所谓“现世”，是上苍赐予我们提升心性的一段时间，是上苍赐予我们磨练灵魂的一个场所。人活着的意义和人生的价值就在
于提升心性，磨练灵魂。简单说，就是这样。

当我看到上面稻盛和夫的回答时，我的心一惊，我顿时觉得我的回答被认同了。或者说，我找到了论据。

为此，我只能感叹，原来人在很多时候，都会自发地、毫无意识地顿悟一些事情！

给JavaScript初学者的鸡汤

纵观现阶段的Web前端开发，诸多的框架，诸多的开发工具，诸多的技术点，如此繁杂，让人应接不暇。随着时间的推移，各种新技术又层出不穷，特别是近两年来，Hybrid这个看似一种”新平台“的兴起，又催生出了一波前端技术潮。我想，很多时候，这些肯定会让人感到恐惧不安，感到疲惫；也会让一个刚进入Web前端开发阵营的开发者感觉无可适从。但我觉得，不要被这些吓到，也不用过度理会这些，这些只不过是由心法衍生出来的招式。要把自己的关注点移到心法上，那心法又是什么？

心法

如果去问开发者，计算机的心法是什么？ 无论他使用哪一种语言，无论从事哪一种平台的开发，我相信他说出来的名词可能是：算法、数据结构，又或者是底层原理、操作系统原理、数据库原理等等；那Web前端开发中的心法又是什么呢？首先，它依然拥有上面的那些名词，但是它也有特定的上层心法：HTML、CSS、JavaScript,当然还有那高深的Webkit。在心法中，JavaScript又是怎么样的角色呢？而它在我心里，是一个利器；它也是接下来所有篇章的重点。

JavaScript

以上两段内容的铺垫只是为了引出JavaScript这个杀手锏；如同武侠小说，当武林高手出场的时候，总得弄出点什么来衬托，比如：美女。

JavaScript是什么

如同C语言、Java语言，它也是一种计算机编程语言；但是它和C、Java不同，它是一种解释型的脚本语言

JavaScript运行环境

一句话就能概括它的运行环境：拥有JavaScript解释器的平台。我敢保证：你的个人PC上肯定具有JavaScript的运行环境，只要你装过浏览器。现阶段，JavaScript的运行环境有这些：各个浏览器（IE、Firefox、Chrome、Safari、QQ浏览器等等）、node.js等等

JavaScript使用范围

  1. 所有的动态网站（我想，现有很少能看到纯静态的网站。夸大点说，你每天能看到的网站，99%的概率使用了JavaScript）
  2. 服务器端（node.js搭建的server）

JavaScript的社区

可以这么说，网上关于JavaScript的资料遍地都是，只要你想学，只要你去找，很容易就找出一堆资料。废话不多说，看github上关于JavaScript的资料：https://github.com/search?utf8=%E2%9C%93&q=javascript

JavaScript的框架

  1. jQuery
  2. Zepto
  3. vue
  4. Angela.js
  5. ……

JavaScript框架如此之多，妈妈再也不用担心你学JavaScript了，对不对。呵呵

总结

从上面的内容，我想你应该能得出以下结论：

  1. 精通Web前端开发，必须精通JavaScript
  2. JavaScript语言的调试环境非常方便
  3. 不愁找不到JavaScript学习资料
  4. 遇到问题，不愁找不到大牛帮忙解答
  5. 想提高生产率，不愁找不到一个合适的JavaScript框架

一次淘宝购物引发出对产品的思考

背景

不久前，在淘宝上买了一个书架，自己动手组装了一下，我说说自己的一点感受：我作为一个小白用户，可能不是太小白，因为是有组装经验的;但当我拿到此款产品的时候，完全没有头绪，也完全不知道从何开始，感觉一团乱。

思考

淘宝上卖书架，卖同类商品的那么多，如何在繁多的商家中脱颖而出，必须找到和其他商家的差异化，找到自己的核心竞争力。那差异化在哪儿呢？我想到几点：

  • 一分钱一分货，人们都知道这个道理，人们也都有想着能以最少的钱拿到最优质的产品和服务；所以对于我们商家，必须是最小成本保证商品的质量和细节（木材的优质可能无法提高，因为成本放在这；但是可以尽量提高烤漆的质量，提高烤漆的等等细节）

  • 淘宝上卖商品，还有个差异化的地方就是：服务，任何地方都一样；海底捞能这么火，这么成功，卖的就是火锅，但是为什么这么成功——差异化的服务。那对于我们商家，差异化的服务又在哪儿呢？听我如下分析：

诉我不才，可能动手能力不是太强；作为一名互联网产品开发，我先说说此款书架的问题：

  • 我买的ABCD四个规格的柜子，店家寄回来的就是一摊木材，我拿到木材毫无头绪，因为不知道哪一块木头对应哪一个规格的柜子，为此，我觉得可以分四个包裹，ABCD分别对应一个包裹，这样对小白用户绝对有利

  • 当根据ABCD分成四个包裹后，大类区分完成了，但是还有一个问题：不知道哪一个木板对应哪一个区域，所以，最好给每一个木板做一个标记（可以简单的标记为1、2、3…）

  • 当有了标记后，得出一份稍微详细的组装说明书，这个说明书不能只是简单地说明组装步骤，有几个重要的方面需要说明：

    • 几个小部件的重要性，必须在说明书突出，因为一些小白可能不知道它的重要性以及组装顺序的合理性
    • 告诉用户，哪些关键点必须注意，否则会影响安装质量，有点东西安装错了，会导致木板损坏，很难恢复（例如：塑料螺帽安装到木板的小孔中的时候，不能砸得太狠，必须保证跟木板齐平，突出来会让木板衔接不密封，太深了会让螺丝无法卡在旋转螺帽中）

总结

不亲身体验自己的产品，不身临其境地体验自己的产品，你根本不知道自己的产品的问题。知道用户在想什么，用户需要什么，才是“用户至上”的意义

七牛云存储上部署iOS应用

Step1
